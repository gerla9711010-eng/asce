# KEIS 廣告上架自動化

把粉專已發布的物件自動同步上 KEIS 廣告追蹤平台。**不存帳密在程式裡** — 你手動登入一次，session 寫進 `profile/` 資料夾，之後跑就自動帶。

## 用法

```bash
# 第一次（或 session 過期）— 開瀏覽器手動登入一次
python publish.py --login

# 之後每次上架
python publish.py YC1868650
python publish.py YC1868650 --headed   # 看瀏覽器跑（debug 用）
```

## 第一次設定

```bash
cd scripts/keis
python -m venv .venv
source .venv/bin/activate    # Windows: .venv\Scripts\activate
pip install -r requirements.txt
playwright install chromium

cp .env.example .env
# 編輯 .env 填 Notion token（KEIS 帳密不用填，下一步手動登入）

python publish.py --login
# 瀏覽器跳出 KEIS 登入頁 → 你手動登入 → 關掉瀏覽器 → session 存好
```

## 之後跑

```bash
python publish.py YC1868650
```

腳本會：
1. 從 Notion 撈該物件的「來源連結」+「粉專貼文連結」
2. 開無頭瀏覽器（用 `profile/` 內 session 自動登入 KEIS）
3. 點新增廣告 → 自動填表 → 送出
4. 成功 PATCH Notion `KEIS同步 = 已同步`
5. 失敗截圖存 `keis_error_<YCxxx>.png`

session 過期會提示你重跑 `--login`。

## 已知會卡的地方

- **欄位 selector**：用了 `get_by_label("帳號")` 這種通用寫法，跑不準再用 DevTools 查實際 selector 調
- **「自動填入」按鈕等 3 秒**：寫死 `wait_for_timeout(3000)`，KEIS 慢的話可能要拉長
- **「新上架」確認文字**：腳本用文字 match，若 KEIS UI 字串不同要改

## 之後可以怎麼擴

- 包成 FastAPI webhook 部到 Railway，n8n 從 LINE 觸發 `上架 KEIS YC1868650` 就自動跑
  - 雲端跑的話 session 持久化會比較複雜（要把 profile/ 上傳到雲端 / 用 cookies 匯出匯入）
- 或包成 yc-ad skill 內建工具，skill 第 4b 步驟直接 call 腳本，不再產操作指令包

# 公買搶單 grab.py

掃「查詢公買 → 買屋需求列表」，把還能申請（`status=Available`）且符合條件的最新名單自動
申請私買，搶到後拿到**沒遮罩的真實姓名＋電話**，推一筆到 LINE。**純 HTTP，不需要瀏覽器。**

## ⚠️ 必須跑在門市網路（店裡電腦）

KEIS 公買功能**「僅限門市內使用」，伺服器擋 IP** — 雲端、家裡、手機都用不了，只有門市網路能用。
所以這支腳本要放在**店裡、連門市網路、且一直開著的電腦**上跑（例如公司電腦不關機）。
腳本啟動會先打 `check-ip` 確認；不在門市網路會直接 LINE 告訴你、不會空跑。

## 用法

```bash
python grab.py                  # 單次 dry-run：列出「這次會搶誰」，不送出
python grab.py --apply          # 單次實搶
python grab.py --watch          # 常駐監控(dry-run)：全天分層掃，只印不搶
python grab.py --watch --apply  # 常駐監控 + 實搶（正式用這個）
```

帳密放 `.env`（`KEIS_USERNAME` / `KEIS_PASSWORD`），腳本用 API 登入拿 JWT、過期自動重登。
**第一次先 dry-run**（`python grab.py`），確認列出的名單符合預期（縣市、類型對不對），再開 `--apply`。

## 搶快：watch 常駐監控

名單**每天早上批次釋出**（規律見下方「釋出時間觀測」），釋出後全店同事集中搶，幾分鐘內掃掉一大半；晚上（實測 19:2x~19:5x）也觀察到同業活躍潮。所以 watch 模式改成**全天分層輪詢**（`WATCH_TIERS`），不再有「時段外完全不看」的空窗：

- 熱門時段（早上開盤 07:30-10:00、晚上同業活躍 17:30-24:00）→ 每 ~5 秒（帶隨機抖動 ±3s）掃一次，逮到符合條件的新名單**立刻搶 + 推 LINE**
- 一般時段（白天 10:00-17:30）→ 1 分鐘一次
- 深夜（00:00-07:30）→ 30 分鐘一次，等同停止監控，純安全網（2026-07-22 使用者拍板調整邊界）
- 配額用完 → 改純觀測上架時間（不搶），依當下分層頻率繼續掃
- 每天換日瞬間 → 回查 `my-applications` 對帳補回漏記的，並把前一天戰果寫進本機 `daily_summary.csv`（一天一行）。**LINE 不再天天推跨日結算**（使用者嫌吵）——想查戰果 LINE 打「戰果」隨時查（見下方），系統死活由心跳告警把關
- IP 被擋（離開門市網路）→ 推 LINE 提醒（30 分鐘節流）並持續重試
- **斷網退避**：暫時性錯誤（逾時等）預設每 3~8 秒重試；連續失敗 ≥10 次（判斷是真的斷網而非開盤塞車）自動拉長到每 5 分鐘重試一次，並推 LINE 告知；恢復時再推一次（只有斷線警告真的送達過才推）。**每晚 00:00~08:00 是已知門市斷網時段**（`EXPECTED_OFFLINE`，2026-07-16 實測 00:00→07:22 斷 7.4 小時）：這段連不上屬預期行為，不告警、靜默低頻重試
- **心跳**（`KEIS_HEARTBEAT_WEBHOOK`）：每 10 分鐘 POST 一下 n8n 的 `/webhook/keis-heartbeat`（`keis-heartbeat-check` workflow 存進 static data；同一支 workflow 排程每天 10:00~22:00 每 2 小時檢查，超過 2 小時沒心跳才推「⚠ 沒心跳」）。把「電腦被關/程序消失沒人知道」變成主動告警，正常日子零訊息。檢查時段刻意避開每晚斷網

### LINE 打「戰果」隨時查（取代每日推播）

LINE 對工作助理打 `戰果`（或 `今日戰果`）→ router 轉發 `keis-battle-report` workflow → 查 Notion 搶單名單 DB 今天的紀錄 → 回覆筆數＋名單。用 LINE reply 回覆，**不吃 push 月額度**。0 筆時會附註「系統掛掉會另有⚠通知，沒收到=正常運作」。

監控時段／頻率改 `grab.py` 的 `WATCH_TIERS`（`(開始, 結束, 間隔秒數)` 陣列，24h 制、需連續涵蓋一整天）。**改了要重啟 watch 進程才生效**（已在跑的載入的是舊設定）。

### 雙帳號分工

同時掛**薛力瑜＋周珈伊**兩個帳號（各 7 配額、共 14），不撞單。搶到的名單分別記在各自帳號下，`grabbed.csv` 會標是哪個帳號搶的。

## 釋出時間觀測（appearances.csv / page1_track.csv）

`grab.py` 內建三個唯讀觀測機制（不吃配額、不影響搶單邏輯）：

- **上架偵測**（`appearances.csv`）：每天第一輪掃時把當下池子最大 `summary_id` 設為基準，之後冒出更大號的名單就把**第一次出現的時刻**記進來。
- **狀態變化觀測**（`observe_status_changes`，只寫 log 不落檔）：記錄既有名單 `Available ↔ CoolingDown` 的轉換，以及 `app_time`（真實申請時間，非建檔時間）的變化——就算狀態字串沒變、但 `app_time` 換了，代表兩輪之間被別人整批吃掉、我方漏接，log 會印 `⚠ 疑似輪詢間隔漏接`。
- **page1 追蹤**（`page1_track.csv`）：每輪都記一筆當下 page1 最大單號，累積成時間序列，供事後回答「這個時間點最新到哪個單號」。

## 📐 釋出規律（2026-07-23 用 111 筆實測觀測資料校正，這是判斷新舊的依據）

拿 `appearances.csv` 累積的 111 筆「第一次看到某單號」去對建檔時間，規律非常乾淨：

- **釋出時刻只有兩種**：`08:00~08:01`（66 筆）和 `18:00`（45 筆），沒有第三種
- **08:00 那批 = 建檔 + 7 天**（落差 6.58~6.98 天，中位數 6.65）← 主要來源
- **18:00 那批 = 建檔 + 0.58~4.07 天**（提前釋出的批次）
- 也就是說：**第一次出現時，建檔一定在 7 天內**（實測最大 6.98 天）
- 反過來，二手回鍋貨最快 = 建檔 →(最快 0.58 天)→ 釋出 → 被申請 →(7 天冷卻)→ 回鍋
  = **建檔至少 7.58 天前**，實務上多半 14 天以上

⚠️ **但建檔日期不能拿來判斷新舊**（2026-07-23 三帳號交叉觀測推翻）：同樣建檔 07-16 的名單
分散在兩個編號帶——`77192`/`77196` 是 07-20 18:00 才進池（許育禎當天就搶到），
`74251`~`74619` 卻是 07-16 18:00 就進池、躺了 7 天後被同業在 07-23 早上掃走。
用建檔日期判斷會把躺 7 天的舊貨當成新單。上面的落差統計只能當背景知識，不參與判定。

**編號大小也不能單獨用**：同一批進池的編號會橫跨一個區間（74xxx 跟 76xxx 同批進池），
「編號 > 當日最大號」會漏掉批次裡編號較小的成員——舊版 `appearances.csv` 就是這樣整批漏記 74xxx。

另一個關鍵實測：**還沒釋出的名單根本不在 query 結果裡**
（今天查全池 2500 筆，建檔 07-19/07-20/07-21 的一筆都沒有，因為它們 7 天還沒到）。
所以「第一次出現 ＝ 剛釋出」，這是總帳判定成立的前提。

觀察到的規律（截至 2026-07）：

- **釋出 = 建檔日 + 剛好 7 天**，釋出時刻原本壓在 ~08:00–08:01，但也觀察過異常提前到 07:30 前、以及晚上 19:2x~19:5x 同業活躍潮。這正是改成 `WATCH_TIERS` 全天分層輪詢（取代原本單一早上時段）的原因：不管幾點釋出/被搶都接得住。
- 建檔只在平日，**週六日不建檔**（所以那兩天釋出量掛零，隔週平日才有貨）。
- **KEIS 語意注意**：`status=CoolingDown` 不是「等待放行」，是**已被申請、進入 7 天排他倒數**；`status=Available` 才是真的可搶。`query()` 用哪個帳號查，**該帳號自己搶到的名單會從自己的查詢結果消失**（`track_top_id` 已用 `also_ids` 補正，避免 max_id 誤判變小）。
- **2026-07-22 邊界已依使用者實測經驗拍板調整**：早上高頻起跑延到 07:30（釋出實測沒早於此）、晚上高頻提早到 17:30 起、深夜 00:00-07:30 併成一段最低頻安全網。

## ⚠️ 可視範圍：為什麼會「明明符合條件卻整批沒看到」（2026-07-23 修）

事故：07-22、07-23 兩天，同業在 08:03~08:35 掃掉十幾筆完全符合我們條件的名單，
我們的 log 從頭到尾沒出現過那些單號——不是慢、也不是篩選太嚴，是**看不到**。

原因：`query` 是照 `summary_id`（建檔序）由大到小排，**不是照「什麼時候變成可搶」排**。
舊版只抓 `page=1&page_size=20`，等於一個「最新 20 筆」的窗口。池子裡卡著大量 CoolingDown
（含我們自己搶到的、7 天內都還佔位），07-23 釋出那批的名次落在 27~54，永遠擠不進那 20 格。
釋出順序跟單號順序本來就不一致（有的建檔隔天就放、有的滿 7 天才放），所以這是必然會再犯的結構問題。

修法：
1. 每輪窗口 20 → **100**（同一個請求，成本沒變）
2. 每 10 分鐘做一次**全池分頁掃描**（`query_deep()`），名次再深也看得到、也才記得完整
3. 看得到不等於要搶：真正決定搶不搶的是下一節的「真新單」判定

## ♻ 只要真新單、不要二手貨（2026-07-23 加）

**KEIS 的殘酷事實（2026-07-23 實測證明）**：名單被申請後進 CoolingDown 7 天，到期會
**回到池子重新變 Available，而且 `app_time` 被清空**——從 API 的單一快照上，二手回鍋貨
跟全新名單長得一模一樣。

證據：拿我方 8 天前搶到的 53 筆去查現在的池子 → **16 筆以 `Available` + `app_time=None` 躺在裡面**
（＝我們自己七天前搶過的東西，現在看起來像全新的），另外 37 筆的 `app_time` 顯示是「7 天後接手的別人」。
所以池子裡那 1000+ 筆 Available 絕大多數是輪了好幾手的二手貨，**不是存貨**。

**判定方法**：API 分不出來，只能靠自己的紀錄擋 → `inventory.csv` 總帳。
只有滿足以下全部才算真新單、才會去搶：

1. 第一次進我們視野時狀態就是 `Available`（未釋出的名單不在 query 裡 → 第一次出現＝剛釋出）
2. 從沒被我們看過是 `CoolingDown`（＝沒被別人拿過）
3. 不是我方自己搶過的（自己的申請在自己帳號的查詢裡會消失，所以搶到當下就直接蓋章，
   `grabbed.csv` 的歷史紀錄也會在啟動時補蓋）
4. 不是「建立總帳當下就躺在池子裡」的（來歷不明，一律不搶）——**不用建檔日期救**，
   那會把躺 7 天的舊貨誤判成新單。代價是建基準當天等於不搶，隔天起正常。

## 📒 名單編號總帳 `inventory.csv`

放在 `%LOCALAPPDATA%\keis-grab\inventory.csv`（本機、不進 OneDrive，怕又被同步弄壞）。
**池子裡出現過的每一個 `summary_id` 都有一行**，這樣事後可以自己分辨「是漏搶、是沒記錄到、還是本來就不該搶」：

| 欄位 | 說明 |
|---|---|
| `首次看到` / `首次狀態` | 這個編號第一次進我們視野的時間與狀態（判定真新單的關鍵） |
| `來源` | `新出現`＝我們親眼看到它冒出來／`基準快照`＝建總帳當下就在池子裡／`我方歷史搶單` |
| `曾冷卻` | `Y` = 曾被人拿過（含我方），回鍋一律不搶 |
| `最後看到` / `最後狀態` / `app_time` | 最近一次掃到的狀態 |
| `符合篩選` / `不符原因` | 對照當下的篩選條件，不符會寫原因（非手機／排除類型=公寓／預算上限不足…） |
| `我方動作` / `我方帳號` | `搶到` / `略過(二手貨)` / 空白 |

⚠️ **第一次啟動會把當下整池標成「基準快照」全部不搶**（來歷不明），從那之後新冒出來的編號才會被當真新單。
所以換機器 / 刪掉這個檔會重新建基準，當天等於不搶——不要隨便刪。

搶單會自動受**今日配額**（API 回傳的 `new_case_quota_remaining`）夾住，配額用完就停手。
搶到的名單會 append 到 `grabbed.csv`（已 gitignore，含真實個資不會進版控），現在最後多一欄「行政區」（需求區域，來自 API `target_areas`）；同步寫 Notion 也會帶這欄（Notion 側要手動加「行政區」文字型別欄位）。詳細規格見 `docs/keis-grab-hardening-and-filters.md`。

## 背後 API（從 HAR 逆出來的）

```
GET  /api/v1/call-purchase/query?inquiry_type=1&page=1&page_size=20&start_date=...&end_date=...
     → data[].status: "Available"=可申請 / "CoolingDown"=已被申請(7天)
     → meta.new_case_quota_remaining = 今日剩餘配額
POST /api/v1/call-purchase/apply/{summary_id}   （無 body）
     → {"success":true,"data":{"display_name":"賴先生","phone_number":"2852068"}}
```

## 讓它在公司電腦「開機就自動監控」（門市網路）

目標：電腦一開著就自己盯，天天如此，不用手動開。用附的 `run.bat`：

1. 把 `grab.py`、`run.bat`、`.env` 三個放同一個資料夾（例如桌面 `keis`）。
2. **雙擊 `run.bat`** 就會開始監控（黑視窗會顯示 `👁 watch 啟動...`，時段外顯示閒置）。
3. 開機自動跑：`Win + R` → 打 `shell:startup` → Enter 開啟「啟動」資料夾 →
   把 `run.bat` **用右鍵拖進去 → 選「在此建立捷徑」**。以後開機/登入就自動啟動。
